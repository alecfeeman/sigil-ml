## Work Package WP01: ServingMode Enum, CLI Flag, and Conditional Startup (Priority: P0)

**Goal**: Introduce the `ServingMode` enum (`local` / `cloud`), add the `--mode` CLI flag to the `serve` subcommand, thread the mode into the `create_app()` factory, and conditionally skip SQLite/poller/scheduler initialization when mode is `cloud`.
**Independent Test**: Run `sigil-ml serve --mode cloud --port 0` and verify the process starts without attempting to open SQLite. Run `sigil-ml serve` (no flag) and confirm default local behavior is unchanged.
**Prompt**: `/tasks/WP01-serving-mode-enum-cli.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [ ] T001 Create `ServingMode` str enum in `src/sigil_ml/config.py` with values `local` and `cloud`
- [ ] T002 Add mode resolution helper in `src/sigil_ml/config.py`: CLI flag > `SIGIL_ML_MODE` env var > default `local`
- [ ] T003 Add `--mode` argument to the `serve` subcommand in `src/sigil_ml/cli.py` and pass to `create_app()`
- [ ] T004 Refactor `create_app()` in `src/sigil_ml/app.py` to accept `mode: ServingMode` parameter, store on `AppState`
- [ ] T005 [P] Conditional startup: skip `ensure_ml_tables()`, `EventPoller`, and `TrainingScheduler` when mode is `cloud`
- [ ] T006 [P] Add `[cloud]` optional extras group (empty) to `pyproject.toml`

### Implementation Notes
- `ServingMode` uses `str` mixin (`class ServingMode(str, enum.Enum)`) for JSON serialization.
- The module-level `app = create_app()` MUST remain unchanged (defaults to LOCAL) for backward compatibility with `uvicorn sigil_ml.app:app`.
- In cloud mode, `AppState` skips `load_models()` entirely -- models are loaded lazily per-tenant in WP05.
- The CLI must use `uvicorn.run("sigil_ml.app:create_app", factory=True, ...)` pattern or pass mode through an env var bridge.

### Parallel Opportunities
- T005 and T006 can proceed concurrently once T001-T004 are in place.

### Dependencies
- None (starting package).

**Requirements Refs**: FR-001, FR-002, FR-006, FR-008

### Risks & Mitigations
- Breaking the existing `app = create_app()` import used by uvicorn: mitigate by defaulting mode to `local` when no argument is supplied.
- CLI-to-factory bridge: uvicorn string import (`"sigil_ml.app:app"`) cannot pass arguments. Solution: set `SIGIL_ML_MODE` env var before `uvicorn.run()`, read it in `create_app()`.

## Work Package WP02: Stateless Prediction Endpoints (Priority: P0)

**Goal**: Make all five `/predict/*` endpoints work in cloud mode by accepting features directly in the request body. Guard endpoints that should not be available in cloud mode (`/train`). Update `/status` to return cloud-appropriate response.
**Independent Test**: Start in cloud mode, POST features to each `/predict/*` endpoint, receive valid prediction responses. Verify `/train` returns 405. No SQLite file exists on the machine.
**Prompt**: `/tasks/WP02-stateless-prediction-endpoints.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [ ] T007 Refactor `/predict/stuck` in `src/sigil_ml/routes.py`: require `features` in body when cloud mode, return 400 for `task_id`-only requests
- [ ] T008 Refactor `/predict/suggest` in `src/sigil_ml/routes.py`: require `classified_events` in body when cloud mode, remove `state.poller._buffer` fallback
- [ ] T009 Refactor `/predict/duration` in `src/sigil_ml/routes.py`: require `features` in body when cloud mode, return 400 for `task_id`-only requests
- [ ] T010 [P] Audit `/predict/quality` in `src/sigil_ml/routes.py`: verify no hidden SQLite path (already requires features in body)
- [ ] T011 [P] Guard `/train` endpoint: return 405 Method Not Allowed with descriptive error in cloud mode
- [ ] T012 [P] Update `/status` endpoint: return mode info and model availability without SQLite queries in cloud mode

### Implementation Notes
- The key change: in cloud mode, the `extract_stuck_features(config.db_path(), task_id)` and `extract_duration_features(config.db_path(), task_id)` paths MUST NOT be called. Those functions import `sqlite3` and will fail without a database.
- For `/predict/suggest`, `state.poller` is `None` in cloud mode. The `state.poller._buffer` fallback on line 184 of current `routes.py` would raise `AttributeError`.
- Each endpoint must retain the existing `task_id` lookup path when mode is `local` -- zero regression.
- Pass `state.mode` (from `AppState`) into handler logic to branch behavior.

### Parallel Opportunities
- T010, T011, and T012 are independent of each other and can proceed in parallel.
- T007, T008, T009 share a common pattern but modify different endpoint handlers.

### Dependencies
- Depends on WP01 (needs `ServingMode` on `AppState` to branch behavior).

**Requirements Refs**: FR-002, FR-003, FR-006, FR-008

### Risks & Mitigations
- Accidentally breaking local mode: each endpoint must retain the existing `task_id` lookup path when mode is `local`.
- The `/predict/suggest` endpoint name maps to model name `"suggest"` in `ml_predictions` -- ensure both paths are covered.

## Work Package WP03: Tenant Context Extraction and Routing (Priority: P1)

**Goal**: Implement `TenantContext` dataclass and a FastAPI dependency that extracts tenant identity from the `X-Tenant-ID` request header in cloud mode. Wire tenant context into all `/predict/*` route handlers. Return 401 when the header is missing in cloud mode; skip tenant check entirely in local mode.
**Independent Test**: Send a request with `X-Tenant-ID: tenant-abc` header, verify the tenant ID is accessible in the route handler. Send a request without the header in cloud mode, verify a 401 error. Send a request without the header in local mode, verify it works (sentinel context used).
**Prompt**: `/tasks/WP03-tenant-context-middleware.md`
**Estimated prompt size**: ~380 lines

### Included Subtasks
- [ ] T013 Create `TenantContext` dataclass in `src/sigil_ml/tenant.py` with `tenant_id: str` and `tier: str = "default"`
- [ ] T014 Implement FastAPI dependency function `get_tenant_context()` that reads `X-Tenant-ID` header from request
- [ ] T015 Wire tenant dependency into all `/predict/*` route handler signatures in `src/sigil_ml/routes.py`
- [ ] T016 [P] Return 401 when tenant header is missing in cloud mode; return sentinel `TenantContext(tenant_id="local", tier="local")` in local mode
- [ ] T017 [P] Make header name configurable via `SIGIL_TENANT_HEADER` env var (default `X-Tenant-ID`)

### Implementation Notes
- `TenantContext` is a simple dataclass -- NOT a Pydantic model. It is internal routing context, not a request/response schema.
- The FastAPI dependency needs access to `AppState.mode` to decide whether to enforce the header. Two approaches: (a) create a closure that captures `state`, or (b) store mode on `app.state` and read from `request.app.state`.
- Do NOT introduce authentication. The API gateway handles that. This is purely header extraction.
- In local mode, the dependency returns a sentinel context but it is never used for model routing (WP05 handles local-mode model resolution via the existing `state.stuck`, `state.workflow`, etc.).

### Parallel Opportunities
- T016 and T017 are independent and can proceed concurrently.

### Dependencies
- Depends on WP01 (needs mode awareness on `AppState`).

**Requirements Refs**: FR-004

### Risks & Mitigations
- Header name convention: document `X-Tenant-ID` and make configurable for flexibility.
- Dependency injection ordering: the tenant dependency must run before the route handler body, which FastAPI handles naturally via `Depends()`.

## Work Package WP04: Model Cache with TTL and LRU Eviction (Priority: P1)

**Goal**: Implement an in-memory `ModelCache` class that stores loaded model weights keyed by `(tenant_id, model_name)` with configurable TTL-based expiration and LRU eviction at a maximum size. This is pure infrastructure -- no integration with endpoints yet.
**Independent Test**: Instantiate `ModelCache`, put a model, get it back, wait past TTL, confirm it reports a miss. Exceed max size, confirm LRU eviction.
**Prompt**: `/tasks/WP04-model-cache-ttl.md`
**Estimated prompt size**: ~400 lines

### Included Subtasks
- [ ] T018 Create `ModelCache` class in `src/sigil_ml/cache.py` with `get()` / `put()` / `evict()` / `evict_all()` interface
- [ ] T019 Implement TTL expiration logic using `time.monotonic()` timestamps per entry
- [ ] T020 Implement LRU eviction when cache exceeds `max_size` (default 100 entries)
- [ ] T021 [P] Add configurable TTL via `MODEL_CACHE_TTL_SECONDS` env var (default 300s) and `MODEL_CACHE_MAX_SIZE` (default 100)
- [ ] T022 [P] Add cache statistics tracking: hits, misses, evictions (exposed via `stats()` method)
- [ ] T023 [P] Thread-safety: wrap `_entries` dict access with `threading.Lock`

### Implementation Notes
- Cache key: tuple `(tenant_id, model_name)` where `model_name` is one of `"stuck"`, `"suggest"`, `"duration"`, `"activity"`, `"quality"`.
- Values: loaded model objects (e.g., `StuckPredictor` instances). Stored in a `ModelCacheEntry` internal dataclass with `model` and `loaded_at` (monotonic timestamp) fields.
- `get()`: check if entry exists and TTL has not expired. If expired, remove and return `None` (count as eviction).
- LRU: on `put()`, if at capacity, remove the entry with the oldest `loaded_at` timestamp.
- `evict(tenant_id)`: remove ALL entries for a given tenant. `evict_all()`: clear entire cache.
- Pure Python, no external dependencies. ~80 lines of implementation.

### Parallel Opportunities
- T021, T022, and T023 can all proceed concurrently once the core cache structure (T018-T020) is defined.

### Dependencies
- None (standalone utility module), but consumed by WP05.

**Requirements Refs**: FR-010

### Risks & Mitigations
- Memory growth: LRU eviction at max_size prevents unbounded growth.
- Stale model serving: TTL prevents indefinite staleness; configurable for operator tuning.
- Race conditions: `threading.Lock` protects all dict operations. Uvicorn's async workers use a thread pool, making this necessary.

## Work Package WP05: Tenant-Aware Model Loading and Fallback Predictions (Priority: P1)

**Goal**: Create the `ModelLoader` protocol and `FilesystemModelLoader` stub, integrate `ModelCache` + `ModelLoader` into `AppState` for cloud mode, update each `/predict/*` handler to resolve models via cache in cloud mode, and implement rule-based fallback predictions for all five model types when no trained model exists for a tenant.
**Independent Test**: Send prediction requests for two different tenants, verify each gets responses from their respective cached models. Send a request for an unknown tenant, verify a rule-based fallback response is returned instead of an error.
**Prompt**: `/tasks/WP05-tenant-model-loading-fallback.md`
**Estimated prompt size**: ~500 lines

### Included Subtasks
- [ ] T024 Create `ModelLoader` Protocol in `src/sigil_ml/loader.py` with `load(tenant_id: str, model_name: str) -> Any | None`
- [ ] T025 Implement `FilesystemModelLoader` in `src/sigil_ml/loader.py`: loads from `{models_dir}/{tenant_id}/{model_name}.joblib`
- [ ] T026 Integrate `ModelCache` + `ModelLoader` into `AppState` initialization for cloud mode in `src/sigil_ml/app.py`
- [ ] T027 Update each `/predict/*` handler to resolve model via `state.model_cache` + `state.model_loader` when in cloud mode
- [ ] T028 [P] Implement rule-based fallback predictions for all five model types matching existing fallback values in `routes.py` and `poller.py`
- [ ] T029 [P] Add structured logging for model load events: cache hit, cache miss + loaded, cache miss + fallback

### Implementation Notes
- `ModelLoader` is a `typing.Protocol` with a single `load()` method. This aligns with Feature 003 (Model Storage Abstraction) which will provide an S3 implementation.
- `FilesystemModelLoader` uses `joblib.load()` to deserialize from `{models_dir}/{tenant_id}/{model_name}.joblib`. Returns `None` if file does not exist.
- Fallback predictions must match existing values: `StuckResponse(probability=0.5, confidence="weak")`, `DurationResponse(estimated_minutes=60.0, confidence_interval=[30.0, 90.0])`, etc. Reference `poller.py` lines 113-131 for exact fallback values.
- In local mode, `AppState` continues to use the existing `load_models()` approach unchanged -- `state.model_cache` and `state.model_loader` are `None` in local mode.
- Per-key locking in cache to prevent thundering herd on concurrent cache misses for the same `(tenant_id, model_name)`.

### Parallel Opportunities
- T028 and T029 can proceed concurrently once the integration wiring (T024-T027) is in place.

### Dependencies
- Depends on WP04 (ModelCache), WP03 (TenantContext), WP02 (stateless endpoints).

**Requirements Refs**: FR-004, FR-005, FR-009, FR-010

### Risks & Mitigations
- Model format mismatch between tenants: log the scikit-learn version on model load for debugging. Fail gracefully with fallback.
- Thundering herd on cache miss: use a per-key lock to prevent concurrent loads for the same model.
- `joblib.load()` on corrupted files: wrap in try/except, log error, return `None` (trigger fallback).

## Work Package WP06: Cloud-Aware Health, Status, and Observability (Priority: P2)

**Goal**: Update `/health` and `/status` endpoints to reflect cloud-mode state accurately. Add `mode` field to `HealthResponse`, report loaded tenants and cache statistics from `/status`, and add per-tenant request counters.
**Independent Test**: Start in cloud mode, call `GET /health`, verify response includes `mode: "cloud"` and does not reference SQLite or poller. Call `GET /status`, verify it lists loaded tenants and cache statistics.
**Prompt**: `/tasks/WP06-health-status-observability.md`
**Estimated prompt size**: ~380 lines

### Included Subtasks
- [ ] T030 Add optional `mode` field to `HealthResponse` schema in `src/sigil_ml/routes.py`
- [ ] T031 Update `/health` handler to report cloud-specific state: mode, model availability per loaded tenant, no poller references
- [ ] T032 Rewrite `/status` handler for cloud mode: list loaded tenants, model versions, cache stats (no SQLite queries)
- [ ] T033 [P] Add per-tenant request counters (in-memory `dict[str, int]`, reset on restart) on `AppState`
- [ ] T034 [P] Add `mode` field to startup log message and structured log output

### Implementation Notes
- `HealthResponse` gains `mode: str | None = None` (optional with default `None` for backward compatibility).
- In cloud mode, `/health` should report: `mode: "cloud"`, uptime, model availability. Do NOT reference SQLite, poller, or `ml_cursor`.
- In cloud mode, `/status` should NOT call `sqlite3.connect()`. Instead report: cache statistics from `state.model_cache.stats()`, loaded tenants, request counts from `state.request_counters`.
- In local mode, both endpoints remain exactly as they are today -- no changes to existing behavior.
- Per-tenant request counters: simple `dict[str, int]` incremented in each `/predict/*` handler when in cloud mode. Exposed via `/status` response.

### Parallel Opportunities
- T033 and T034 are independent of each other and of the endpoint handler changes.

### Dependencies
- Depends on WP05 (model cache + tenant loading wired up), WP01 (mode awareness).

**Requirements Refs**: FR-007

### Risks & Mitigations
- Adding `mode` field to `HealthResponse` could break consumers expecting exact schema: use optional field with `None` default.
- Adding new keys to `/status` response: cloud response is entirely different from local response, but local response remains unchanged.

## Dependency & Execution Summary

- **Sequence**: WP01 (foundation) -> WP02 + WP03 + WP04 (parallel wave) -> WP05 (integration) -> WP06 (polish)
- **Parallelization**: After WP01 is complete, WP02, WP03, and WP04 are independent and can be worked in parallel by separate agents. WP05 integrates their work. WP06 is final polish.
- **MVP Scope**: WP01 + WP02 constitute the minimal "cloud mode starts and serves predictions" milestone. Adding WP03 + WP04 + WP05 enables multi-tenant model resolution. WP06 adds operational polish.
